<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → RevenueCat SDK | Payment processing via StoreKit — RevenueCat is trusted intermediary |
| Client → EntitlementManager | Local state derived from CustomerInfo — can be stale but not spoofable for server-side checks |
| iOS paywall → App Store | StoreKit purchase flow — Apple validates receipts server-side |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-52-01 | S (Spoofing) | EntitlementManager local state | accept | Server-side RLS enforcement (Phase 51) is the hard gate; local state is UX-only |
| T-52-02 | T (Tampering) | PaywallView purchase flow | mitigate | RevenueCatUI uses StoreKit directly; receipt validation by RevenueCat server-side; no custom purchase logic to tamper with |
| T-52-03 | I (Information Disclosure) | Subscription status in Settings row | accept | Subscription tier is not PII; visible only to authenticated device owner |
| T-52-04 | D (Denial of Service) | Network failure during purchase | mitigate | RevenueCatUI shows error state; restore purchases available; EntitlementManager trusts cached CustomerInfo offline (D-05 from Phase 50) |
| T-52-05 | E (Elevation of Privilege) | Client-side entitlement bypass | accept | Server-side RLS (Phase 51 SUB-08/SUB-09) prevents API-level bypass; client display is cosmetic |
</threat_model>
